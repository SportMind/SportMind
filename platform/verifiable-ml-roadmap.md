# Verifiable ML Roadmap — SportMind

**What cryptographically provable sports intelligence would mean, and how to get there.**

This is a forward-looking planning document. SportMind v3.x uses a rules-based
reasoning framework — structured intelligence applied by an LLM, not a trained
neural network. Verifiable ML applies to neural network inference specifically.
This document maps the path from the current architecture to one where signals
can carry cryptographic proofs.

It is not a build item for the current library. It is a documented trajectory
so future contributors and integrators understand the direction of travel.

---

## What verifiable ML actually means

```
CURRENT STATE (v3.x):
  An agent loads SportMind skill files + calibration data
  Applies structured reasoning (6-step chain)
  Produces a signal: direction + SMS + modifiers
  
  TRUST MODEL: You trust that:
    (a) the skill files are accurate (MIT licensed, auditable)
    (b) the calibration records are genuine (all public, pre-submitted)
    (c) the agent applied the framework correctly (LLM reasoning)
    
  The trust is human and social — not cryptographic.

VERIFIABLE ML STATE (v4.x target):
  A trained SportMind model produces a signal
  A ZK proof is generated: "this output came from SportMind-v4 applied to
  these inputs, committed on-chain at block N before the match"
  Anyone can verify the proof without re-running the model
  
  TRUST MODEL: Cryptographic — mathematical proof, not social trust.
  Verification is permissionless and requires no trust in the operator.
```

## Why this matters for the Chiliz ecosystem specifically

```
FAN TOKEN HOLDER PERSPECTIVE:
  Current: "I trust this SportMind signal because the calibration records are public"
  Verifiable: "This signal is mathematically proven to have been generated by
               SportMind-v4 before the match, using the inputs committed on-chain"
  
  The second is a fundamentally stronger trust model for on-chain applications.
  A fan token holder acting on a signal that carries a ZK proof can verify
  independently that the signal was generated honestly.

ON-CHAIN PREDICTION MARKET PERSPECTIVE (Azuro integration):
  A SportMind signal submitted to an Azuro pool could carry a ZK proof of provenance
  This allows the protocol to verify signal quality without trusting the operator
  Creates the foundation for on-chain signal markets — where signal quality
  (measured by calibration accuracy) commands a price premium
  
DeFi COLLATERAL PERSPECTIVE (Phase 5 RWA):
  If SportMind signals are used as inputs to DeFi protocols (collateral assessment,
  yield calculation), ZK-verified provenance becomes a security requirement
  A protocol accepting a "SportMind says this token is Phase 3 active"
  assertion would need cryptographic proof in a trust-minimised system

THE COMMERCIAL CASE:
  Verifiable signals would command a premium over unverified signals
  Calibration history (96%, 126+ records) becomes provably attached to the model
  SportMind transitions from "trusted intelligence layer" to "verified intelligence infrastructure"
```

## Technical path

```
STEP 1: Train a SportMind signal model (precondition)
  
  Current: rules-based framework (LLM applies markdown skill files)
  Required: trained neural network that can be proven
  
  WHAT TO TRAIN ON:
    Input features:
      - Competition tier (encoded from sport domain skills)
      - Athlete modifier values (from athlete intelligence skills)
      - Macro modifier (from macro state)
      - DSM level (from disciplinary intelligence)
      - Historical form scores
      
    Training data:
      - 126+ calibration records (current; need 500+ for reliable training)
      - Community-contributed records (priority for v3.x community drive)
      
  CALIBRATION THRESHOLD:
    Minimum for first trained model: 500 records across 10+ sports
    Target for production model: 2,000+ records, 21+ sports
    Current status (v3.42): 126 records — in training data accumulation phase
    
  IMPORTANT: The rules-based framework (v3.x) will remain the primary mode
  until the trained model achieves equivalent or better calibration accuracy.
  Community calibration records generated under v3.x are training data for v4.x.

STEP 2: Select ZK framework
  
  PRIMARY CANDIDATE: EZKL (ezkl.xyz)
    - Converts ONNX neural network models to ZK circuits
    - Supports Ethereum/EVM proof verification
    - Active development; growing sports/prediction use cases
    
  ALTERNATIVE: Giza (gizatech.xyz)
    - Focuses on on-chain ML inference via Cairo (StarkNet)
    - Strong for prediction market use cases
    
  CHILIZ CHAIN CONSIDERATION:
    Chiliz Chain is EVM-compatible (Chain ID 88888)
    EZKL proofs verify on EVM chains — Chiliz Chain compatible
    StarkNet/Cairo (Giza) would require bridge for Chiliz Chain settlement
    
  RECOMMENDATION: EZKL for primary development (EVM + Chiliz compatibility)

STEP 3: Generate and verify proofs
  
  PROOF GENERATION:
    SportMind signal request received
    Model runs inference: input features → signal output
    ZK proof generated: "these inputs produced this output via SportMind-v4"
    Proof + output committed on-chain (or to verifiable data store)
    
  PROOF VERIFICATION:
    Anyone queries the proof with the claimed inputs and output
    Verifier contract confirms proof is valid
    Verification does not require access to model weights
    
  LATENCY CONSIDERATION:
    Current ZK proof generation: seconds to minutes per inference
    Pre-match signals (T-48h): latency not an issue
    Real-time signals (T-2h): may require proof generation infrastructure
    
    Recommendation: pre-generate proofs for scheduled pre-match windows;
    T-2h signals use standard (unproven) for now

STEP 4: On-chain signal registry
  
  A Chiliz Chain contract that:
    → Accepts SportMind v4 signals with ZK proofs
    → Verifies proof validity
    → Records signal provenance on-chain
    → Makes signals queryable by external protocols (Azuro, DeFi protocols)
    
  This is the "SportMind Signal Registry" — a verifiable on-chain record of
  every signal ever generated, auditable by anyone.
```

## What this means for current SportMind contributors

```
CALIBRATION RECORDS ARE THE CRITICAL INPUT:
  Every calibration record submitted under v3.x is training data for v4.x.
  The community calibration drive is not just about improving current accuracy —
  it is building the dataset that makes verifiable ML possible.
  
  Priority record types for v4.x training data:
    → Records across diverse sports (non-football gaps are largest)
    → Records that test specific modifiers (dew_factor, morning_skate, etc.)
    → Records with full input capture (all modifier values documented)
  
  See: FIRST-RECORD-CHALLENGE.md for how to contribute

CURRENT ACCURACY IS THE BENCHMARK:
  v4.x trained model must match or exceed v3.x calibration accuracy
  Current benchmark: 96% direction accuracy (121/126 records)
  A trained model that achieves <90% would not replace the rules-based framework
  
  This means: the rules-based framework is not a temporary measure.
  It is the production architecture until a trained model proves equivalent quality.
```

## Timeline estimate

```
v3.x (current — through 2026):
  Rules-based framework
  Community calibration drive (target: 500+ records)
  Social/LunarCrush connectors operational
  
v4.0 (2027 target):
  First trained SportMind signal model (if 500+ calibration records achieved)
  EZKL ZK proof integration
  Chiliz Chain signal registry contract
  Pilot: verifiable signals for Tier 1 football fan tokens only
  
v4.x (2027-2028):
  Expand verifiable signals across sports
  On-chain signal market (signals with provenance command price premium)
  Full Azuro integration with ZK-verified signal provenance
  RWA/SportFi Phase 5 applications using verified signals as DeFi inputs

DEPENDENCY: This entire roadmap depends on reaching 500+ calibration records.
The community calibration drive (see FIRST-RECORD-CHALLENGE.md) is the
critical path item for SportMind's long-term architecture.
```
